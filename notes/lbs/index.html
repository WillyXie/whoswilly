The Little Book of Semaphores

This note serves as a personal study guide and summary of The Little Book of Semaphores by Allen B. Downey [1][2]. It outlines key synchronization patterns and constructs detailed in the book to provide a quick, accessible reference.

Preface

The author wrote the book to address standard operating system curricula that present synchronization as a single module with fixed solutions, leaving students unable to solve novel concurrency problems independently. Developed during his teaching at Wellesley College—using hands-on puzzles—and refined at Olin College into a dedicated synchronization course, the book is designed to build a deep intuition for synchronization. The intention is to teach synchronization as a set of modular, composable patterns that can be understood in isolation and combined to solve complex concurrency problems.

Chapter 1: Introduction to Synchronization

1.1 Non-Determinism of Program Execution

To reason about concurrent software and synchronization, we require a clear model of how computer programs run. In the simplest execution model, computers execute instructions one by one in sequence. The order of events is entirely deterministic and can be inferred directly from the program text: statement A will always execute first if it is written before statement B.

However, this straightforward ordering breaks down when parallel computation and multi-threading are introduced. During execution, different physical cores run instructions simultaneously, and multiple threads are scheduled dynamically and unpredictably by the operating system. Consequently, it becomes impossible to determine whether a statement on one processor/thread executes before, after, or at the exact same time as a statement on another. This unpredictable timing making program state unpredictable is known as non-determinism.

Parallel Computation

A hardware configuration where a computer features multiple processors or physical cores operating simultaneously, allowing different execution paths to run in true physical concurrency at the exact same time.

Multi-Threading & Threads

A software abstraction where a thread represents a sequential flow of instructions. On a single core, multiple threads are interleaved dynamically (context-switched) by the scheduler, creating the illusion of parallel execution.

A few different program puzzles are demonstrated to show the consequences of non-determinism. Each one shows a simple multi-threaded program whose output or final state cannot be predicted in advance but depends on the order in which the statements are executed, which is called the execution path.

Puzzle 1 — Who Prints First

When two threads both call print function, the operating system can interleave their output at any point or make them executed by different cores at the same time. Even individual characters of a single string may be split across context switches, producing garbled results that are impossible to anticipate. 

// Thread A            // Thread B
print("yes")           print("no")

// Possible outputs:
//   "yesno"
//   "noyes"
//   "ynoes" <= characters interleaved mid-write

Puzzle 2 — Concurrent Writes

When two threads assign different values to the shared variable, the final result depends entirely on which thread's write happens last. There is no way to determine this order from the program text alone. 

int x = 0;

// Thread A            // Thread B
x = 5;                 x = 7;
print(x)

// Final value of x is NON-DETERMINISTIC:
//   Terminal shows 5 if Thread A updates last or print first before B updates
//   Terminal shows 7 if Thread B updates last

Puzzle 3 — Concurrent Updates

A compound statement like count += 1 is not a single hardware instruction—it compiles into a read, an arithmetic operation, and a write. If two threads interleave across these three steps, one thread's increment can be silently overwritten, so the counter ends up smaller than expected. This class of bug is called a race condition. 

int count = 0;

// Thread A             // Thread B
temp_a = count;         temp_b = count;
temp_a++;               temp_b++;
count = temp_a;         count = temp_b;

// Expected: count == 2
// Actual:   count == 1
Shared Variables

For most of time variables are local, meaning they belong to a single thread. There tend to be few synchronization problems, because threads doesn't interact upon them. But some variables are shared among different threads for interaction; this is one of the way they interact with each others and can cause issues if they are not synchronized properly.

Race Conditions

The outcome of execution depends on which thread wins the "race" to read/write, potentially yielding incorrect, inconsistent states.

1.2 Synchronization

In common use, synchronization implies making two or more events occur at the exact same time. In computer systems, Synchronization refers to enforcement of relationships among events—covering any number of events and any relationship (before/during/after).

Programmers manage these relationships using synchronization constraints, which define the required execution order of events. The two most fundamental constraints are:

  • Serialization: Event A must happen before Event B (e.g., a producer must write data before a consumer reads it).
  • Mutual Exclusion: Event A and Event B must not happen at the same time (e.g., two threads must not write to the same shared variable simultaneously).

Serialization - The Lunch Puzzle & Message Passing

Imagine the case that Alice and Bob live in different cities, and Alice wants to guarantee that she eats lunch before Bob today without a synchronized clocks to coordinate. The solution is simple: Alice tells Bob, "Do not eat lunch until I call you." Alice eats lunch, and only when she is finished, she calls Bob.
In computer systems, this call acts as a message passing mechanism. Event A (Alice's lunch) is guaranteed to complete before Event B (Bob's lunch) starts because Bob's execution thread is blocked (waiting) until the message arrives.

Mutual Exclusion - Nuclear Reactor Puzzle

Imagine you and Bob monitor a reactor from remote stations. You can both take lunch breaks, but only when you never do so simultaneously. In this case, you'll need to call to make sure another one is not having a break before you can do so.
Recall the Concurrent Updates example: to ensure correctness, we must make uninterruptible read-modify-write steps. While some computer architectures provide atomic instructions, software systems typically rely on synchronization primitives—like semaphores—to control concurrent access to shared variables.

Atomic Instruction

An instruction that executes as a single, indivisible step. Once it begins, it runs to completion without interruption or interleaving from other processors/threads, preventing other concurrent execution paths from observing any intermediate states.

Chapter 2: Semaphores

2.1 Definition

Semaphore was first proposed by Edsger Dijkstra[3] as a synchronization primitive to manage shared resource access.[4] It acts as a protected integer value that can only be controlled through operations with the following attributes:

  1. Restricted Interface: When you create a semaphore, you can initialize its value to any integer. After initialization, you can only increment or decrement it. You cannot read the current value of the semaphore directly.
  2. Blocking on Decrement: When a thread decrements the semaphore, if the result is negative, the thread immediately blocks itself through notifying the scheduler. It cannot continue executing until another thread increments the semaphore.
  3. Unblocking on Increment: When a thread increments the semaphore, if there are other threads waiting, one of the waiting threads gets unblocked and is marked ready to run.

Depending on whether the internal counter is, the value of a semaphore conveys the exact state of the system:

  • Positive: Represents the number of threads that can perform a decrement operation without blocking (equivalent to available resource).

  • Zero: Means there are no threads waiting in the blocked queue, but the next thread that attempts to decrement will block immediately.

  • Negative: Represents the exact number of threads that are currently blocked and waiting in the queue.

2.2 Operation Terminology

Semaphores have been implemented across many different operating systems and languages, different terms are used to describe the increment and decrement operations:

  • P / V: Dijkstra's original Dutch notation (<P> for proberen - "to test/try", and V for verhogen - "to increment/raise").
  • decrement / increment: Describe what the operations do upon the value.
  • wait / signal: Describe what they are used for.

2.3 Why Semaphores?

Semaphores are highly favored in concurrent programming because they impose deliberate constraints that help programmers avoid synchronization errors. By hiding the internal counter and restricting operations to atomic increments and decrements, they provide a clean, organized abstraction that makes it easy to reason about and demonstrate the correctness of concurrency solutions.
Furthermore, semaphores are supported most of the modern systems, they provide efficient block-and-wake solutions both highly performant and portable across different architectures.

Chapter 3: Basic Synchronization Patterns

These patterns represent the fundamental building blocks of concurrency. By learning these idioms, you can combine them to solve more complex synchronization problems without resorting to ad-hoc logic.

3.1 Signaling

Specialty: The simplest synchronization pattern. One thread signals that a certain event or statement has executed, releasing a second thread waiting on that signal.

The Issue: Thread B must execute statement B2 only after Thread A has executed statement A1. If run concurrently without synchronization, the order is non-deterministic and B2 can execute first.

The Hint: Initialize a semaphore to 0. Because the starting value is zero, any thread that calls wait() on it will block immediately until another thread calls post() to increment it. 

// Initialization
sem = Semaphore(0)

// Thread A                 // Thread B
statement A1                sem.wait()
sem.post()                  statement B2

// Deterministic Result:
// statement A1 is GUARANTEED to execute before statement B2.

3.2 Rendezvous

Specialty: Two-way signaling. Two threads are guaranteed to meet at a common execution point before either is allowed to proceed further.

The Issue: Thread A must not execute A2 until Thread B has executed B1. Simultaneously, Thread B must not execute B2 until Thread A has executed A1.

The Hint: Initialize two independent semaphores, aArrived and bArrived, both to 0. Each thread signals its own arrival first, then waits on the other thread's arrival. 

// Initialization
aArrived = Semaphore(0)
bArrived = Semaphore(0)

// Thread A                 // Thread B
statement A1                statement B1
aArrived.post()             bArrived.post()
bArrived.wait()             aArrived.wait()
statement A2                statement B2

// Deterministic Result:
// statement A1 executes before statement B2, and
// statement B1 executes before statement A2.

3.3 Mutual Exclusion (Mutex)

Specialty: Protecting access to a shared mutable state (critical section) so that at most one thread can execute it at a time.

The Issue: When multiple threads access and update the same shared variables concurrently (e.g., compound increments like count += 1), their operations can interleave, causing race conditions and lost updates.

The Hint: Initialize a semaphore (typically called a mutex) to 1. A value of 1 represents a single available resource token (permission to enter). The first thread to call wait() acquires it, and subsequent threads will block until the active thread releases it by calling post(). 

// Initialization
mutex = Semaphore(1)

// Thread A                 // Thread B
mutex.wait()                mutex.wait()
// critical section         // critical section
// (e.g., count += 1)       // (e.g., count += 1)
mutex.post()                mutex.post()

// Deterministic Result:
// Threads execute critical sections sequentially. No race condition occurs, 
// and count is updated correctly.

3.4 Multiplex

Specialty: A generalization of the Mutex pattern. It limits the number of threads that can enter a critical section simultaneously to a maximum of N.

The Issue: Resource throttling. Without limits, too many concurrent threads accessing a resource (such as a database pool or a network socket) can exhaust system capacity.

The Hint: Initialize the semaphore to the capacity limit N. Up to N threads can decrement the semaphore without blocking; the $(N+1)$-th thread will block until one of the active threads exits and increments the semaphore. 

// Initialization
multiplex = Semaphore(N)

// Thread i
multiplex.wait()
// critical section (at most N threads active simultaneously)
multiplex.post()

// Deterministic Result:
// At any point in time, the number of threads executing inside 
// the critical section is at most N.

3.5 Barrier

Specialty: Generalizing the Rendezvous pattern to n threads. No thread is allowed to cross the barrier to its post-barrier statements until all n threads have arrived.

The Issue: Coordinating phases in parallel algorithms (e.g. game loops, matrix multiplication) where all threads must finish computing step t before any can start step t+1.

The Hint: Keep a counter of arrived threads protected by a mutex. Use a semaphore (initially 0) as a gate (turnstile). The last thread to arrive unlocks the turnstile, allowing all threads to proceed sequentially. 

// Initialization
mutex = Semaphore(1)
barrier = Semaphore(0)
count = 0

// Thread i (out of n threads)
mutex.wait()
  count = count + 1
mutex.post()

if count == n:
  barrier.post()

barrier.wait()
barrier.post() // Turnstile: wakes up the next waiting thread

// Deterministic Result:
// All n threads complete their pre-barrier statements before any thread 
// is allowed to execute its post-barrier statements.

3.6 Queue (Exclusive/Rendezvous Queue)

Specialty: Pairing threads of different types (e.g., matching leaders and followers in a ballroom dance, or clients and servers).

The Issue: Threads must wait until a partner thread of the opposite type is ready, matching them up in a strict 1-to-1 lockstep.

The Hint: Use two separate queues (semaphores initialized to 0): a leaderQueue and a followerQueue. When a leader arrives, it signals that it is ready for a follower, then waits on the leader queue. The follower does the reciprocal. 

// Initialization
leaderQueue = Semaphore(0)
followerQueue = Semaphore(0)

// Leader Thread            // Follower Thread
followerQueue.post()        leaderQueue.post()
leaderQueue.wait()          followerQueue.wait()
dance()                     dance()

// Deterministic Result:
// Leaders and followers are paired up in lockstep. A leader and a follower 
// will always execute dance() together.

Chapter 4: Classical Synchronization Problems

These are the historic puzzles that laid the groundwork for modern concurrency theory. They test the limits of basic patterns and introduce advanced synchronization ideas like turnstiles, counters, and resource constraints.

4.1 Producer-Consumer Problem

Specialty: Coordinating data exchange between two active execution threads (producers and consumers) using a shared, bounded buffer array.

The Issue: If the producer runs too fast, it will overwrite unread slots (buffer overflow). If the consumer runs too fast, it will read duplicate or empty slots (buffer underflow). Concurrently updating buffer indices also leads to race conditions.

The Hint: Protect the shared buffer indices with a mutex. Track the number of empty slots using a counting semaphore spaces (initially N, the buffer capacity), and track filled slots using another counting semaphore items (initially 0). 

// Initialization
mutex = Semaphore(1)
items = Semaphore(0)
spaces = Semaphore(N) // N is buffer capacity

// Producer                 // Consumer
spaces.wait()               items.wait()
mutex.wait()                mutex.wait()
// add item to buffer        // remove item from buffer
mutex.post()                mutex.post()
items.post()                spaces.post()

// Deterministic Result:
// The buffer never overflows (producer blocks if spaces == 0) and never 
// underflows (consumer blocks if items == 0). Buffer updates are race-free.

4.2 Readers-Writers Problem

Specialty: Managing read-heavy resources. Allows multiple threads to read concurrently, but requires exclusive access (no other readers or writers) for threads that modify the resource.

The Issue: Writers must have exclusive access to prevent dirty reads or concurrent write conflicts. However, if readers continuously arrive and occupy the resource, writers can be starved indefinitely.

The Hint: Track the active reader count using a shared integer counter. Use a mutex to protect modifications to this counter. The first reader to arrive acquires the write lock (commonly called roomEmpty), blocking all writers. The last reader to leave releases roomEmpty, allowing writers to enter. 

// Initialization
mutex = Semaphore(1)
roomEmpty = Semaphore(1)
readCount = 0

// Reader                   // Writer
mutex.wait()                roomEmpty.wait()
  readCount++                 // write to shared resource
  if readCount == 1:        roomEmpty.post()
    roomEmpty.wait()
mutex.post()

// read shared resource

mutex.wait()
  readCount--
  if readCount == 0:
    roomEmpty.post()
mutex.post()

// Deterministic Result:
// Multiple readers can read simultaneously. Writers have exclusive access 
// (no readers or other writers can be active).

4.3 Dining Philosophers Problem

Specialty: Illustrating resource deadlock and starvation. N philosophers sit around a circular table. Each philosopher needs two adjacent forks (left and right) to eat.

The Issue: If all philosophers simultaneously sit down and pick up their left forks, all right forks are unavailable. Every philosopher blocks waiting for their neighbor's fork, resulting in a permanent deadlock where everyone starves.

The Hint: Limit the number of philosophers at the table simultaneously using a multiplex semaphore initialized to N-1 (e.g. 4 for a 5-person table). This guarantees that at least one philosopher is always able to acquire both forks, eat, and release them. 

// Initialization
forks = [Semaphore(1), Semaphore(1), Semaphore(1), Semaphore(1), Semaphore(1)]
multiplex = Semaphore(4) // At most 4 philosophers at a table of 5

// Philosopher i (0 to 4)
multiplex.wait()
  forks[left(i)].wait()
  forks[right(i)].wait()
  // eat()
  forks[right(i)].post()
  forks[left(i)].post()
multiplex.post()

// Deterministic Result:
// Deadlock is impossible because at least one philosopher is guaranteed 
// to get both forks and finish eating, releasing them.

4.4 Cigarette Smokers Problem

Specialty: Advanced multi-resource coordination. Three smoker threads each possess a single ingredient (tobacco, paper, or matches). An agent puts two ingredients on the table. Only the smoker with the third ingredient can proceed.

The Issue: If smokers grab ingredients directly from the table, they can deadlock (e.g. Smoker A grabs tobacco, Smoker B grabs paper, both wait forever for matches).

The Hint: Do not let smokers access the table directly. Instead, create helper threads called pushers that monitor the ingredients on the table, update boolean flags, and signal the correct smoker's semaphore once all necessary materials are available. 

// Initialization
tobacco = Semaphore(0); paper = Semaphore(0); match = Semaphore(0)
isTobacco = false; isPaper = false; isMatch = false
mutex = Semaphore(1)

// Pushers (run by helper threads when agent puts items)
// Pusher A (runs when tobacco is put on table)
mutex.wait()
  if isPaper:
    isPaper = false; match.post() // wake match smoker
  else if isMatch:
    isMatch = false; paper.post() // wake paper smoker
  else:
    isTobacco = true
mutex.post()

// Smoker with Match (runs when signal on match is received)
match.wait()
  make_cigarette()
  agent.post() // request more ingredients
  smoke()

// Deterministic Result:
// Only the correct smoker is awakened for the available ingredients. 
// No deadlock or starvation occurs.

Chapter 5: Less Classical Synchronization Problems

These puzzles focus on practical scenarios like barber services, tribes refilling communal food pots, and Santa coordinating his helpers. They introduce ideas like balking (leaving if full) and complex state checks.

5.1 Dining Savages

Specialty: Refill coordination. Savages eat from a communal pot of M servings. If the pot becomes empty, the cook must be awakened to refill it, while savages wait.

The Issue: If multiple savages check the pot concurrently, they can attempt to eat from an empty pot, or wake the cook multiple times, causing race conditions.

The Hint: Protect the pot servings count with a mutex. If a savage finds 0 servings left, they signal a potEmpty semaphore to wake the cook and block on a potFull semaphore until the cook refills and signals. 

// Initialization
mutex = Semaphore(1)
potEmpty = Semaphore(0)
potFull = Semaphore(0)
servings = 0

// Savage Thread                        // Cook Thread
while true:                             while true:
  mutex.wait()                            potEmpty.wait()
  if servings == 0:                       servings = M // refill
    potEmpty.post()                       potFull.post()
    potFull.wait()
  servings--
  // eat()
  mutex.post()

// Deterministic Result:
// Savages only eat when servings > 0. The cook is awakened exactly when 
// servings reach 0, refilling the pot before savages proceed.

5.2 Barbershop Problem

Specialty: Balking pattern. Customers arrive for a haircut. If the shop is full, they leave (balk) immediately instead of waiting in line.

The Issue: If the barber is asleep, customers must wake them. If the barber is busy, customers wait. If all waiting chairs are full, newly arriving customers must leave without blocking.

The Hint: Track the number of customers in the shop. If it reaches the chair capacity limit, subsequent customer threads return immediately (balk). Otherwise, they increment the count and wait on the barber. 

// Initialization
mutex = Semaphore(1)
customer = Semaphore(0)
barber = Semaphore(0)
customerChair = Semaphore(0)
barberChair = Semaphore(0)
customers = 0

// Customer Thread                      // Barber Thread
mutex.wait()                            while true:
if customers == n:                        customer.wait()
  mutex.post()                            barber.post()
  balk() // leave if chairs full          // cut_hair()
customers++                               barberChair.wait()
mutex.post()                              customerChair.post()

customer.post()
barber.wait()
// get_haircut()
customerChair.post()
barberChair.wait()

mutex.wait()
customers--
mutex.post()

// Deterministic Result:
// Customers leave if all n waiting chairs are full. Barber sleeps when 
// customers == 0, and wakes up to cut hair sequentially.

5.3 Santa Claus Problem

Specialty: Multi-barrier logic. Santa sleeps in the North Pole and is only awakened when either all 9 reindeer return from vacation, or when 3 elves have questions. Reindeer have priority.

The Issue: Santa must only help elves in groups of 3, or coordinate toy delivery with all 9 reindeer. Elves and reindeer must wait in their respective lines without mixing.

The Hint: Use counters for elves and reindeer protected by a mutex. When the count of reindeer reaches 9, or the count of elves reaches 3, they wake Santa. Santa then uses respective semaphores to release them in batches. 

// Initialization
santaSem = Semaphore(0); reindeerSem = Semaphore(0); elfSem = Semaphore(0)
mutex = Semaphore(1)
elves = 0; reindeer = 0

// Santa Claus Thread
while true:
  santaSem.wait()
  mutex.wait()
  if reindeer == 9:
    reindeer = 0
    reindeerSem.post() // deliver toys (all 9 reindeer)
  else if elves == 3:
    elfSem.post() // help 3 elves
  mutex.post()

Chapter 6: Not-So-Classical Problems

These problems deal with complex real-world data structures (like databases with search, insert, and delete priorities) and gender-segregated facility concurrency.

6.1 Search-Insert-Delete

Specialty: Tri-party mutual exclusion. Inserters can run concurrently with searchers but not other inserters. Deleters need exclusive access to the list.

The Issue: Standard reader-writer locks only support two access levels. In this problem, inserters are a middle category that can tolerate searchers but must block other inserters and deleters.

The Hint: Use the Lightswitch pattern twice: once for searchers (to block deleters) and once for inserters (to block deleters). In addition, use a mutex to ensure inserters run one at a time. 

// Initialization
insertMutex = Semaphore(1)
noSearchers = Semaphore(1)
noInserters = Semaphore(1)
searchSwitch = Lightswitch()
insertSwitch = Lightswitch()

// Searcher Thread                      // Inserter Thread
searchSwitch.lock(noSearchers)          insertMutex.wait()
// search()                             insertSwitch.lock(noInserters)
searchSwitch.unlock(noSearchers)        // insert()
                                        insertSwitch.unlock(noInserters)
// Deleter Thread                       insertMutex.post()
noSearchers.wait()
noInserters.wait()
// delete()
noInserters.post()
noSearchers.post()

// Deterministic Result:
// Searchers run concurrently with each other and Inserters. Inserters run 
// sequentially. Deleters have exclusive access (no searchers/inserters active).

6.2 Unisex Bathroom

Specialty: Conditional mutual exclusion with multiplexing limits. Men and women can use the same bathroom, but not at the same time. At most N people can be inside.

The Issue: If a man is inside, no woman can enter, and vice versa. However, other men can enter up to a limit of N, and the bathroom must be freed when the last person leaves.

The Hint: Implement two Lightswitches (one for men, one for women) that attempt to acquire the lock to the bathroom (empty). Combine this with gender-specific semaphores initialized to N to cap the occupancy. 

// Initialization
empty = Semaphore(1)
maleSwitch = Lightswitch()
femaleSwitch = Lightswitch()
maleMultiplex = Semaphore(3) // max 3 men

// Male Thread
maleSwitch.lock(empty)
maleMultiplex.wait()
  // use_bathroom()
maleMultiplex.post()
maleSwitch.unlock(empty)

// Deterministic Result:
// The bathroom is never shared by men and women simultaneously, and at 
// most 3 people of the same gender are inside at any time.

Chapter 7: Not Remotely Classical Problems

This chapter covers advanced batch processing patterns where threads must wait in line and are only admitted when the previous batch completely finishes and leaves.

7.1 Sushi Bar Problem

Specialty: Batching and clearing pattern. A bar has 5 seats. If you arrive and there is space, you sit. If it is full, you must wait until all 5 people currently eating finish and leave.

The Issue: If customers continue to arrive and sit one-by-one as soon as a single seat opens, waiting customers who arrived earlier could wait indefinitely.

The Hint: Maintain a boolean flag mustWait. If the bar is full (5 eating), set mustWait = true. Arriving threads block on a queue. When the active eating count drops to 0, release waiting threads in batches of up to 5 and reset the flag. 

// Initialization
mutex = Semaphore(1)
block = Semaphore(0)
eating = 0; waiting = 0
mustWait = false

// Customer Thread
mutex.wait()
if mustWait:
  waiting++
  mutex.post()
  block.wait()
else:
  eating++
  mustWait = (eating == 5)
  mutex.post()

// eat_sushi()

mutex.wait()
eating--
if eating == 0:
  n = min(5, waiting)
  waiting -= n
  eating += n
  mustWait = (eating == 5)
  block.post() // release waiting batch
mutex.post()

Chapter 8: Synchronization in Python

Python provides several built-in synchronization primitives within the threading module. This section summarizes how they correspond to semaphore concepts:

  • threading.Lock: A binary lock (mutex) that can be acquired or released.
  • threading.Semaphore: A standard counting semaphore initialized to a given capacity.
  • Context Managers: Python locks and semaphores support the with statement, which automatically acquires the resource and guarantees its release even if exceptions occur.
import threading

# Create a lock (mutex) and a semaphore
mutex = threading.Lock()
sem = threading.Semaphore(3)

# Thread execution using context managers
with mutex:
    # critical section (automatic acquire and release)
    pass

Chapter 9: Synchronization in C

In C, synchronization is typically handled using the POSIX Threads (pthreads) library and the POSIX Semaphore API:

  • pthread_mutex_t: The standard mutex type, initialized statically or dynamically.
  • sem_t: The counting semaphore type defined in <semaphore.h>, controlled via sem_wait() and sem_post().
  • pthread_cond_t: Condition variables used alongside mutexes to wait for specific conditions to become true.
#include <pthread.h>
#include <semaphore.h>

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
sem_t sem;

void* thread_func(void* arg) {
    pthread_mutex_lock(&mutex);
    // critical section
    pthread_mutex_unlock(&mutex);
    return NULL;
}
